<h2>v0.13.3 — Selection drag, live-resize reflow, font-thicken actually works</h2>
<p>A round of "core terminal interactions you'd expect to just work" — three real bugs that had been silently broken, plus a sweep of theme selection colors so picking a theme doesn't render selected text unreadable.</p>
<ul>
  <li><b>Drag-to-select is fixed.</b> Three things were stacking against it: SwiftTerm's <code>allowMouseReporting</code> defaults to TRUE (so mouseDragged bails the moment any TUI app touches mouse mode), the <code>font</code> setter unconditionally calls <code>selectNone()</code> (so any re-applied appearance wiped the selection), and worst of all <code>resizeSubviews</code> unconditionally set <code>selection.active = false</code> on every layout pass — including the sub-pixel ones SwiftUI re-renders trigger, which fired between every <code>mouseDragged</code> event during a drag. Now: <code>allowMouseReporting = false</code> in <code>makeNSView</code> (selection always wins over apps); font setter only fires when the font actually differs; SwiftTerm patched to NOT clear selection on resizeSubviews (the <code>processSizeChange</code> path already clears when the cell grid genuinely changes, which is the only case where the clear is correct). Drag, double-click, triple-click, shift-extend, ⌘C — all responsive on the first try.</li>
  <li><b>Window resize now reflows TUI apps live, not at the end of the drag.</b> The previous build coalesced live-resize ticks (skipped SwiftTerm's per-tick <code>terminal.resize</code> during a drag, fired one SIGWINCH on release) for a perf win. Side effect: Claude / fzf / vim rendered their output at the OLD width while the window had already grown wider, producing visible wrap mismatches. Now every <code>setFrameSize</code> tick goes straight through <code>processSizeChange</code>, so the cell grid + SIGWINCH stay in lockstep with the visible width through the entire drag. SwiftTerm 1.13's reflow is fast enough per-tick that the drag stays smooth.</li>
  <li><b>Font Thickness slider actually changes glyph weight now.</b> The SwiftTerm patch that adds <code>fontThicken</code> support was setting the <code>strokeWidth</code> attribute AFTER caching the attribute dict — so only the first frame of each glyph-attribute combination drew thick (then the cached no-stroke version was returned for every subsequent draw, instantly reverting to thin). Patch reordered to set strokeWidth before the cache write. Sliding Thickness 0 → 1.5 now boldens text live.</li>
  <li><b>Theme selection colors don't fight foreground anymore.</b> SwiftTerm renders the selection background at FULL alpha and draws foreground glyphs on top, so the selection color has to contrast with the theme's foreground. Four themes were broken:
    <ul>
      <li><b>Dracula</b>: canonical <code>(68,71,90)</code> "Current Line" gray was only ~3 luminance points off the bg — selection visually disappeared. Switched to Dracula's "Comment" blue <code>(98,114,164)</code>.</li>
      <li><b>Gruvbox</b>: yellow accent <code>(250,189,47)</code> washed out the cream foreground. Switched to Gruvbox <code>bg2</code> dark <code>(80,73,69)</code> — same color vim's <code>:hi Visual</code> uses.</li>
      <li><b>Ayu Dark</b>: bright orange <code>(255,180,84)</code> fought cream text. Switched to Ayu's ide-selection blue <code>(64,127,170)</code>.</li>
      <li><b>GitHub Light</b>: full-saturation dark blue <code>(9,105,218)</code> under near-black text was unreadable. Switched to GitHub's pale-blue selection tint <code>(208,222,243)</code>.</li>
    </ul>
  </li>
  <li><b>Native AppKit context menu on the terminal view.</b> Replaced the SwiftUI <code>.contextMenu</code> wrapper (which installed an NSGestureRecognizer that interfered with mouseDown→mouseDragged) with <code>view.menu = NSMenu(...)</code> set directly on the SwiftTerm NSView. Copy / Paste / Select All target the responder chain so SwiftTerm's own validation auto-disables them when appropriate; Find / Clear / New Tab / Splits route through a singleton dispatcher to the existing notifications.</li>
  <li><b>Pane focus tracked at the NSEvent layer.</b> Replaced the SwiftUI <code>.onTapGesture</code> on each pane (which was holding mouseDown hostage for tap-vs-drag disambiguation) with a window-scoped <code>NSEvent.addLocalMonitorForEvents(.leftMouseDown)</code>. The monitor walks the responder chain to find the clicked <code>TermyTerminalView</code> and updates <code>activePaneId</code>, then returns the event unchanged so SwiftTerm receives the full mouseDown stream.</li>
</ul>
